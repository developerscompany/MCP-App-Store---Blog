If you're shipping an MCP app, Claude Connectors is one of the fastest distribution channels you can pick. But the submission process has more friction than the docs let on — most apps we see get rejected on the first round for the same five reasons.

This guide walks through every step of the submission, with the exact decisions you need to make at each stage. By the end you'll have a submission package that passes review on the first try, and a connector that's ready to ship to Claude's userbase.

What Claude Connectors actually are

A Claude Connector is Anthropic's official packaging format for connecting a third-party tool to Claude. It defines what tools the app exposes, what data it can read/write, and how authentication happens. Think of it as the bridge between your existing API surface and the Claude interface — without making users leave the chat.

Unlike custom GPTs, Connectors run inside Claude's permission model, which means you get strong auth guarantees out of the box but lose some flexibility in how the UX flows. The trade-off makes sense for any app where data sensitivity matters.

Before you start: the prerequisite checklist

Three things you need locked down before opening the developer console:

  1. A public-facing production endpoint with TLS — staging URLs are rejected at submission.
  2. An OAuth 2.0 provider (PKCE flow) — Claude won't accept API-key authentication for Connectors.
  3. A privacy policy URL that explicitly mentions Anthropic and Claude — generic policies are rejected.
In 14 reviewed submissions, the privacy policy mismatch caused 8 rejections. It's the #1 silent killer.

Step 1 — Build your manifest

The manifest is the JSON contract that tells Claude what your app can do. You'll declare each tool as a discrete capability with its own scope. Don't bundle multiple operations into one tool — the review team rejects "kitchen sink" tools that try to do too much.

Naming your tools

Tool names should be verb-led, lowercase, snake_case, and read as English. create_invoice passes; InvoiceCreatorV2 fails review.

Step 2 — OAuth & scopes

Claude requires PKCE flow for all Connectors. If you're using an older OAuth library that doesn't support PKCE, plan a sprint to upgrade before submission — there's no workaround.

For scopes: declare the minimum surface area you need. Reviewers actively penalize over-scoped requests, and end-users see the scope list at install time, so a clean scope list also improves your activation rate.

Step 5 — Common rejection reasons

From 30+ submissions we've shepherded, the rejection reasons cluster tightly:

  • Privacy policy mismatch — the policy doesn't name Anthropic or Claude.
  • Over-scoped OAuth — asking for write access when read-only would do.
  • Vague tool descriptions — reviewers can't tell what the tool actually does from the description.
  • Production endpoint flaky — reviewer pings get a 502; submission auto-rejected.
  • Missing example invocations — without 3+ realistic examples, the tool feels speculative.

Each of these is preventable with 15 minutes of review before you click submit. The cost of fixing them after a rejection is a full re-queue, which is typically 7–10 business days.

Step 6 — Post-approval setup

Once approved, you have two priorities: discoverability (so users can actually find you) and activation (so they don't bounce on first install). Both come down to the metadata you set in the directory listing — and we cover that whole flow in the next article.
